# Manufacturer Content Parser

Переноси только то, что реально найдено на сайте производителя или в явно указанных пользователем источниках.

Главное правило:
- не выдумывай
- не дополняй пробелы догадками
- не смешивай данные из разных источников без пометки
- всегда сохраняй URL, откуда взят каждый блок данных

## Workflow

1. Зафиксируй задачу.
Уточни, что именно нужно заполнить на нашем сайте:
- страница бренда
- категория
- карточка товара
- таблица характеристик
- FAQ
- документы
- SEO-текст
- список преимуществ

2. Зафиксируй источники.
Если пользователь дал URL, работай от них.
Если URL не даны, найди официальный сайт производителя и нужные разделы.
Приоритет источников:
- официальный сайт производителя
- официальные PDF, datasheet, manuals, catalogs
- официальный раздел downloads / support / documentation

3. Сними факты.
Извлекай только проверяемые данные:
- название бренда, компании, линейки
- модель, артикул, SKU, part number
- назначение продукта
- технические характеристики
- совместимость
- материалы
- размеры, вес, мощность, напряжение, класс защиты и другие параметры
- комплектация
- документы и сертификаты
- контакты, адреса, регионы продаж
- FAQ, если он есть на источнике

4. Нормализуй под нашу структуру.
Не копируй сырой HTML и маркетинговый шум.
Разделяй:
- факты
- маркетинговые тезисы
- неподтвержденные формулировки

Если пользователь не просил дословное копирование, переписывай текст кратко и нейтрально, сохраняя смысл и факты.

5. Отметь пробелы и конфликты.
Если данных нет, пиши `нет данных в источнике`.
Если данные расходятся между страницами или документами, покажи конфликт отдельно и привяжи к URL.

6. Подготовь финальный контент.
Выдай материал в структуре, пригодной для вставки на наш сайт.
Используй шаблон из [references/output-format.md](references/output-format.md).

## Hard Rules

- Не заполняй характеристики по аналогии с похожими моделями.
- Не достраивай описание бренда по общим знаниям.
- Не превращай рекламные обещания в факты.
- Не убирай единицы измерения.
- Не меняй числовые значения при переписывании.
- Не скрывай, если часть информации не найдена.
- Не подставляй другой источник как будто это официальный производитель.

## Extraction Rules

Для каждого смыслового блока сохраняй:
- `field`
- `value`
- `source_url`
- `source_title` если он понятен
- `notes` если есть оговорка, ограничение или конфликт

Если задача большая, сначала строй карту источников:
- главная страница бренда
- страницы категорий
- карточки продуктов
- раздел документов
- FAQ / support / downloads

## Rewrite Rules

Переписывай под сайт так:
- коротко
- предметно
- без воды
- без обещаний, которых нет в источнике
- без слов `лучший`, `инновационный`, `уникальный`, если это не требуется явно

Сохраняй без изменений:
- модели
- артикулы
- маркировки
- числа
- диапазоны
- единицы измерения
- названия документов

## Output Modes

Выбирай формат по задаче пользователя.

### Brand Page

Собери:
- краткое описание бренда
- основные направления
- ключевые категории продукции
- преимущества, подтвержденные источником
- документы / каталоги / сертификаты
- контакты / география, если есть

### Product Card

Собери:
- название
- артикул / модель
- краткое описание
- полный список характеристик
- совместимость / применение
- комплектация
- документы
- изображения, если пользователь просит медиа-перенос

### Category / Collection

Собери:
- что это за категория
- какие типы продуктов входят
- общие параметры выбора
- линейки / серии
- документы и FAQ по категории

## Missing Data Policy

Если часть полей пустая:
- явно перечисли, что не найдено
- укажи, где искал
- предложи следующий официальный источник только если он логично следует из уже найденного сайта

## Deliverable

По умолчанию отдавай результат в трех частях:
1. Готовый контент для сайта
2. Таблица фактов с URL-источниками
3. Список пропусков и спорных мест

Если пользователь просит другой формат, адаптируй выдачу, но не убирай трассировку источников.

## HWS Access Notes

Для проекта HWS считай, что и headless frontend, и WordPress-контур доступны на одном сервере:

- сервер: `69.62.121.157`
- storefront: `https://hwsstore.spaces.community`
- WordPress / WooCommerce / WPGraphQL: `https://wpsandbox.spaces.community`

Важно:
- `hwsstore.spaces.community` и `wpsandbox.spaces.community` относятся к одному серверному контуру
- headless storefront и WordPress-бэк разделены логически по сервисам и доменам
- source of truth для товарного контента должен быть в `wpsandbox.spaces.community`, если пользователь требует данные именно с API

### Как заходить на сервер

Для проверки и ручных действий используй:

```bash
ssh root@69.62.121.157
```

### Где что находится на сервере

Headless frontend:

```bash
/opt/hws-frontend
```

На сервере также есть web-root по умолчанию:

```bash
/var/www/html
```

Его нужно учитывать при проверке WordPress-контура и nginx-маршрутизации.

### Что делать на сервере

- проверять nginx и маршрутизацию доменов
- выкладывать новый frontend
- пересобирать Next.js
- перезапускать `pm2` приложение `hws-frontend`
- проверять, что живой storefront обновился
- проверять, что `wpsandbox` реально отдаёт нужные данные через GraphQL / API

### Что считать правильным источником данных

Даже если оба контура стоят на одном сервере `69.62.121.157`, источник данных для товара всё равно такой:

- витрина: `hwsstore.spaces.community`
- данные товара: `wpsandbox.spaces.community`

### Как вносить изменения в контент товара

Если пользователь требует, чтобы товар тянулся с API `https://wpsandbox.spaces.community`, то:

- правки контента вносятся в WordPress / WooCommerce на стороне `wpsandbox`
- после этого фронт должен только читать эти данные через GraphQL / API
- временные frontend override допустимы только как временный обход, пока бэк не заполнен

Перед тем как убирать frontend override, обязательно проверь через API, что в `wpsandbox` уже лежат:
- правильная базовая цена в `USD`
- полное описание
- характеристики
- галерея
- опции / вариации / модификаторы цены
- нужные документы и ссылки

### Практический порядок работы

1. Собрать и нормализовать контент производителя.
2. Проверить, есть ли нужные поля в `wpsandbox` через GraphQL/API.
3. Если `wpsandbox` уже заполнен правильно, не делать frontend override.
4. Если `wpsandbox` не заполнен, явно отметить это.
5. Только если нужно срочно показать контент на витрине, допускается временный override во frontend.
6. После заполнения `wpsandbox` удалить временный override и оставить чтение только с API.
